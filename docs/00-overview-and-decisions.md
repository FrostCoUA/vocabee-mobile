> Вступний розділ документації **Vocabee**. Тут зібрано призначення продукту, два стани користувача, глосарій, константи економіки та підсумковий журнал затверджених рішень (D1–D11).
>
> **Позначки в усій документації:** `[ЗАРАЗ]` — поведінка в поточному коді; `[НОВЕ]` — затверджена зміна / новий функціонал; `Припущення (уточнити)` — поведінка, яка ще потребує підтвердження.

## 1. Призначення продукту

**Vocabee** — застосунок для вивчення іноземних слів. Користувач читає тексти (книги, статті, субтитри, дописи) і прямо «на ходу» зберігає незнайомі слова: вводить їх текстом або диктує голосом, а застосунок одразу підтягує AI-переклад. Зібрані слова не лежать купою — їх можна впорядкувати у тематичні словники (наприклад «серіал», «подорож», «робота»), кожен зі своєю парою мов, кольором та іконкою. Так замість абстрактного списку слів користувач отримує живу персональну колекцію, прив'язану до власного контексту читання.

Друга половина циклу — тренування. На основі збережених слів Vocabee дає вправи на запам'ятовування, щоб слово перейшло з «бачив колись» у активний словниковий запас. Продукт побудований навколо бджолиної тематики: внутрішня валюта — **монетки (beecoins)** з іконкою стільника / Sparkle. **[НОВЕ за D11]** Пошук безкоштовний, а монетками оплачуються понадлімітні словники та кожен конкретний варіант перекладу, збережений у словник. Безкоштовний пробний режим дозволяє спробувати продукт без акаунта, а повна економіка (монетки, реклама, промо) вмикається після входу через Google.

### Сигнатурний цикл

```
Читаю → зберігаю слово (текст / голос + AI-переклад) → організовую в тематичні словники → тренуюсь
```

| Крок | Що відбувається |
|---|---|
| **Читаю** | Користувач натрапляє на незнайоме слово під час читання / перегляду. |
| **Зберігаю слово** | Вводить слово текстом або диктує голосом (STT); Vocabee підтягує AI-переклад. |
| **Організовую** | Слово потрапляє у тематичний словник (тема, пара мов, колір, іконка). |
| **Тренуюсь** | Збережені слова використовуються у вправах на запам'ятовування. |

## 2. Два стани користувача

Vocabee розрізняє **два стани**: `anonymous` (без акаунта) та `authenticated` (вхід через Google). Реєстрація — це **момент апгрейду**: саме після входу вмикаються монетки, реклама та промо.

> **Ключове правило (D2):** анонімний користувач **НЕ має монеток**. Він користується безкоштовним пробним режимом із жорсткими лімітами. У коді `addBees` / `spendBees` роблять early-return, якщо `!isAuthenticated`.

| Аспект | `anonymous` (без акаунта) | `authenticated` (Google) |
|---|---|---|
| **Доступ до основного функціоналу** | Читання, збереження слів, тренування | Повний доступ |
| **Ліміт словників** | Максимум **2** словники (`FREE_DICTIONARY_LIMIT`) | 2 безкоштовні + платні (по `DICTIONARY_CREATION_BEE_COST` за кожен наступний) |
| **Ліміт слів** | Максимум **50** слів усього (`AnonymousFreeWordLimit`) | Без жорсткого ліміту слів (обмеження — економіка та квоти сервера) |
| **Монетки (beecoins)** | **Немає** — нараховувати/списувати не можна (D2) | Є; стартовий баланс `INITIAL_BEE_BALANCE = 50` |
| **Пошук / збереження перекладу** | Пошук і збереження доступні в межах пробного режиму, **без монеток** | **[ЗАРАЗ, legacy v1]** `TRANSLATION_SEARCH_BEE_COST = 1` за пошук; **[НОВЕ, D11/v2]** пошук 0, кожен збережений варіант за активною політикою (початково −1) |
| **Реклама (rewarded ad)** | **Недоступна** | Доступна: базово `+10` (`REWARDED_AD_BEE_AMOUNT`), верифікована на сервері |
| **Промо / акції** | **Недоступні** | Доступні (config-driven, бонус зверху базової реклами) |
| **Синхронізація з сервером** | Немає — дані лише локально (анонім) | Повна: баланс, словники, слова, тема звіряються із сервером |
| **Зберігання даних** | Локально на пристрої | Сервер — джерело істини; локально — кеш + оптимістичні зміни |

## 3. Глосарій термінів

| Термін | Значення |
|---|---|
| **Монетка (beecoin)** | Внутрішня валюта застосунку (бджолина тематика, іконка Sparkle / стільника). **[НОВЕ за D11]** Оплачує понадлімітні словники та збережені варіанти перекладу; сам пошук безкоштовний. Існує **лише** для authenticated-користувачів (D2). |
| **Словник (topic / dictionary)** | Тематична колекція слів зі своєю парою мов, кольором та іконкою (наприклад «серіал», «подорож»). На бекенді — сутність `topic` із полями `source_lang` / `target_lang`. |
| **Слово (word)** | Окремий збережений елемент: оригінал + AI-переклад, що належить конкретному словнику. |
| **Напрямок перекладу** | Орієнтація пари мов усередині словника (з якої → на яку). Також задає пріоритетну мову розпізнавача мовлення (STT). Зберігається **по словнику** (D8). |
| **Промо / акція** | Кампанія, що дає бонусні монетки за дії (перегляди реклами, стрік, реєстрацію). Повністю описується конфігом із сервера; клієнт лише рендерить (D4). |
| **Стрік (streak)** | Серія днів поспіль із виконанням умови (наприклад «≥3 реклами/день 5 днів поспіль»). Використовується в daily-streak промо. |
| **Лідерборд (leaderboard)** | Рейтинг користувачів за метрикою (наприклад щотижневий топ-10 за переглядами реклами) з винагородою та періодичним скиданням. |
| **Sync-статус** | Стан синхронізації локального елемента із сервером (наприклад очікує заливки, синхронізовано, помилка, pending-delete). |
| **Soft-delete** | «М'яке» видалення на сервері: запис позначається видаленим, але фізично лишається (можливе відновлення, узгодженість синхронізації). |
| **Pending-delete** | Локальний проміжний стан: елемент позначено на видалення, але ще не синхронізовано із сервером (підтримує Undo). |
| **Оптимістичне списання** | Клієнт відразу зменшує баланс / застосовує зміну для миттєвого UX, **не чекаючи** сервера, а потім звіряє результат із авторитетною відповіддю сервера (D1). |

> Шляхи у репозиторії:

|---|---|---|

| `REWARDED_AD_BEE_AMOUNT` | **10** | Базова винагорода за перегляд 1 реклами. Промо нараховуються **зверху** цієї суми. |

## 5. Журнал рішень (D1–D11)

> Це **джерело істини** з продуктових рішень: власник продукту погодив. Кожен пункт — остаточна домовленість із поясненням «чому саме так».

### D1. Економіка — сервер авторитетний
Сервер стає **єдиним джерелом істини** для балансу, усіх списань (понадлімітні словники, пошук перекладу) та лімітів. Клієнт списує **оптимістично** заради миттєвого UX, але звіряє баланс із сервером при кожній відповіді/синхронізації; ендпоінт `applySync` **мусить** валідувати квоти та списувати монетки, відхиляючи або обрізаючи понадлімітне. **Чому:** це усуває поточний баг подвійного списання (зараз монетки знімаються і на клієнті у `VocabeeStore.createTopic`, і на сервері у `topics.service.ts`). Окремо: нарахування за рекламу має стати **верифікованим і ідемпотентним** на сервері (Server-Side Verification / разовий nonce) замість «голого» `POST /wallet/rewarded-ad` — щоб ніхто не накручував монетки.

> Цінова деталь «списання за пошук» у D1 є історичною й **superseded by D11**. Інваріанти server-authoritative, idempotency та серверного enforcement лишаються чинними.

### D2. Анонім — без монеток
Анонімні користувачі **не мають** монеток, реклами та промо. Натомість — безкоштовний пробний режим: максимум 2 словники та максимум 50 слів усього. Монетки/реклама/промо вмикаються **тільки** після входу через Google; реєстрація — момент апгрейду (у коді `addBees`/`spendBees` роблять early-return, якщо `!isAuthenticated`). **Чому:** економіка має сенс лише для ідентифікованого користувача (інакше валюта легко обнуляється/фармиться скиданням застосунку), а пробний режим без валюти — це чесна та проста демонстрація цінності перед реєстрацією.

### D3. [ЗАРАЗ, legacy pricing; superseded by D11] Видалення — без повернення монеток + Undo
**[ЗАРАЗ, legacy pricing]** За видалення платного словника (3-го і далі) монетки **не повертаються**. Натомість для слів і для словників показуємо **Undo** (снекбар ~5–10 с), а сервер робить **soft-delete**, локально — **pending-delete** до синхронізації. **Чому:** неповернення монеток прибирає експлойт «створив платний словник → видалив → повернув монетки → повторив» (фарм), а Undo + soft-delete захищають від випадкових втрат, не відкриваючи цю діру.

> Правило «без повернення» описує історичний стан і **superseded by D11**. Undo, soft-delete та pending-delete лишаються чинними інваріантами.

### D4. Промо — config-driven Promo API
Бекенд повертає список активних промо. Кожна промо несе: `id`, `type`, винагороду (конфігурабельні монетки), прогрес (`current`/`target`), вікно скидання, копірайт банера (`title`/`subtitle`/`icon`/`accent`/`priority`) і вміст ботомшита (заголовок, пояснення, візуалізація прогресу, список CTA-кнопок, у т.ч. опційну кнопку «Дивитись рекламу»). Клієнт лише **рендерить** із конфіга — жодної захардкодженої промо-логіки. Базові `+10` за рекламу лишаються; промо — бонус **зверху**. **Чому:** маркетинг має змінювати кампанії та суми **без релізу клієнта**, а уніфікований контракт виключає розсинхрон між клієнтом і бекендом. Приклади кампаній: (a) milestone «10 реклам → +20», повторюється; (b) daily_streak «≥3 реклами/день 5 днів поспіль → +50»; (c) one_time «реєстрація → +50»; (d) leaderboard «щотижневий топ-10 за переглядами реклами → +50».

### D5. Онбординг — реальний вхід + розумна розвилка
Екран Auth в онбордингу стає **справжнім** Google-входом (зараз бутафорський — усі кнопки просто `onDone`). Розвилка: увійшов **і** на сервері є дані → **пропускаємо** вибір мов, тягнемо мови/баланс/тему з сервера → Main; увійшов, але акаунт **новий** → показуємо вибір мов → пушимо на сервер → Main; **скіпнув** вхід → вибір мов → зберігаємо локально (анонім) → Main. Прапор `hasCompletedOnboarding` зберігається (наступний запуск: Splash → Main). **Чому:** повертати наявного користувача без зайвого перепитування мов — це повага до його часу, а нового або анонімного — провести через мінімально потрібний вибір; усе керується одним станом (увійшов / є дані).

### D6. Мова словника
Пара мов у профілі (Налаштування) = **дефолт** для нових словників; її можна змінити, і це впливає лише на **майбутні** словники. При **створенні** словника пару можна **оверрайднути** саме для нього. Існуючі словники незмінні при зміні профілю — їхня пара **фіксується на момент створення** (бекенд `topics` уже зберігає `source_lang`/`target_lang` на словник). **Чому:** користувач часто вчить кілька мовних пар одночасно; глобальна зміна дефолту не повинна «ламати» вже наповнені словники.

### D7. Теми словників — набір іконок
До словників додаємо набір **іконок** (не лише кольори): серіал/фільм, книга/читання, подорожі (літачок), їжа (виделка+ложка), робота, школа/навчання, спорт/зал, музика, природа/тварини, техніка/IT, шопінг, діти, здоров'я/медицина, загальна (зірка/стільник). Пікер іконок — у шторці створення словника поряд із кольорами. **Чому:** іконка зчитується миттєво й робить список словників наочнішим за самий лише колір, посилюючи тематичну організацію (ключова цінність продукту).

### D8. Напрямок STT
Перемикач напрямку перекладу всередині словника (виставляє пріоритетну мову розпізнавача мовлення STT) **зберігається по словнику**. Зараз він живе лише в пам'яті екрана й скидається при відкритті. **Чому:** напрямок диктування — стала властивість конкретного словника, а не разовий вибір; його скидання щоразу — зайве тертя для користувача, що часто диктує слова.

### D9. Sync-конфлікт при вході — справжній мерж
Коли є **і** локальні анонімні дані, **і** дані на сервері, пробуємо залити локальні словники/слова в серверний акаунт, рахуємо **вартість у монетках** за зайві словники понад наявний/безкоштовний ліміт і **чесно** повідомляємо, скільки спишеться. Варіанти: (1) погодитись на списання → мерж; (2) затерти серверні дані локальними; (3) відкинути локальне, лишити серверне; (4) нічого не робити → скасувати вхід; (запасний) увійти іншим email (дані лишаються окремо). Якщо монеток на мерж **не вистачає** — лишаються лише не-мерж-варіанти. **Чому:** вхід не повинен мовчки знищувати чиюсь роботу (ні локальну, ні серверну); прозорий вибір + чесна ціна — це довіра. **Майбутнє:** функція «мерж акаунтів» (перенесення словників між акаунтами, коли користувач назбирає монетки) — терміни TBD.

### D10. Тренування — гібридний відбір з анти-повтором
Поточний відбір (`shuffle + sortedBy(knowledgePercent).take(10)` + єдине поле `knowledge_percent`) дає відчуття «вчу ті самі 10 слів». Замінюємо на гібрид **пріоритет + Leitner-інтервали + анти-повтор**: бал слова = слабкість × прострочення × частка помилок, зважений рандом; Leitner-бокси (1–5) з інтервалами; **кулдаун** нещодавно показаних + домішування **нових** слів щосесії + випадання вивчених. Оцінка **бінарна** (Знаю / Не знаю); сесія **10** = ≈7 повторення + ≈3 нові. Додаються поля (клієнт + сервер; майбутня `0013_training_fields.sql` **[НОВЕ, ще не створена]**): `timesCorrect`, `timesWrong`, `boxLevel 0..5`, `lastReviewedAtEpochMillis`, `dueAtEpochMillis`. **Чому:** головна вимога власника — щоб тренування не здавалось повторенням одного й того самого; кулдаун + зважений рандом + нові слова це забезпечують. Деталі — [11-practice-training.md](11-practice-training.md).

### D11. Платимо за збережений переклад; коротке повернення
Пошук перекладів безкоштовний. Кожен конкретний варіант перекладу, який
авторизований користувач зберігає у словник, створює окреме серверне списання
за активною політикою (початково −1). Видалення слова протягом 3600 секунд від
серверного списання повертає точну історичну суму один раз; пізніше повернення
немає. Видалення словника повертає всі ще придатні word-charge та його власний
topic-charge, якщо платний словник теж створено не давніше вікна. Політики
версійні й керуються через client-admin-web; legacy-записи без charge не
повертаються. D11 замінює цінові деталі D1/D3, але не їхні інваріанти
server-authoritative, idempotency, soft-delete та Undo.
