# 11 — Тренування (Practice): флов і алгоритм відбору

> Дотримуємось позначок **[ЗАРАЗ]** — поточний код, **[НОВЕ]** — затверджена зміна.
> Рішення власника: алгоритм обирає асистент, головна вимога — **щоб не здавалось, що вчиш ті самі 10 слів**; оцінка **бінарна** (Знаю / Не знаю); сесія **10 слів** = повтор + нові.

---

## 1. [ЗАРАЗ] Поточна реалізація

Джерело: `App.kt:3327–3530` (`PracticeScreen`, `PracticeSetupScreen`, `PracticeFlipCard`, `PracticeDoneState`), `AdjustWordKnowledgeUseCase` (`VocabularyUseCases.kt:86`), серверна колонка `topic_words.knowledge_percent` (`migration 0006`).

- **Єдине поле знань** — `knowledgePercent: Int (0–100)` і на клієнті (`WordEntry`), і на сервері. **Немає** лічильників правильних/неправильних, **немає** часу останнього повторення, **немає** історії спроб, боксів чи інтервалів.
- `updatedAtEpochMillis` бампиться **будь-яким** редагуванням слова (не лише тренуванням) — як «давність практики» **непридатне**.
- **Відбір сесії** (`App.kt:3376–3381`):
  ```kotlin
  availableCardKeys
      .shuffled(Random(shuffleSeed))
      .sortedBy { (_, card) -> card.word.knowledgePercent.coerceIn(0, 100) }
      .take(10)
  ```
  → рандом, потім **детермінований сорт за knowledge% (найслабші першими)**, перші 10.
- **Оцінка** — 2 кнопки «Не знаю» / «Знаю» (`App.kt:3505–3520`), зсувають `knowledgePercent`. Після промаху — кнопка «Далі».
- **Наслідок проблеми:** доки слабкі слова не «підтягнуться» за %, сорт щоразу витягує **ті самі** найслабші 10 → відчуття «вчу одне й те саме». Давність і помилки не враховуються.

## 2. [НОВЕ] Обраний підхід — гібрид «пріоритет + Leitner + анти-повтор»

Чистий spaced-repetition (SM-2/Anki) — надлишковий для бінарної оцінки і важкий. Чистий «найслабші першими» — причина проблеми. Тому беремо **гібрид**, оптимізований під варіативність:

| Складова | Що дає |
|---|---|
| **Пріоритет-бал** (слабкість + давність + помилки) | слова, які реально треба повторити, мають вищий шанс |
| **Leitner-інтервали** (бокси 1–5) | вивчене слово «дозріває» все рідше → виходить із щоденного пулу |
| **Зважений рандом** замість топ-10 | щосесії вибірка інша, навіть за однакового пулу |
| **Кулдаун нещодавно показаних** | слово з минулої/поточної сесії тимчасово виключається → не лізе вдруге поспіль |
| **Домішування нових** | у кожній сесії кілька ще не тренованих слів → відчуття прогресу |
| **Випадання вивчених** | бокс 5 + високий % з'являються рідко → не «крутяться» постійно |

Саме `кулдаун + зважений рандом + нові слова + випадання вивчених` прибирають відчуття «ті самі 10 слів».

## 3. [НОВЕ] Модель даних — нові поля

Поточний `knowledgePercent` лишається (для відображення кільця/прогресу), але для розумного відбору додаємо поля. Усі — і на клієнті (`WordEntry`, `WordEntity`), і на сервері (`topic_words`), синхронізуються (D1, сервер авторитетний).

### Клієнт — `WordEntry` / `WordEntity`

| Поле | Тип | Призначення |
|---|---|---|
| `knowledgePercent` | Int 0–100 | [ЗАРАЗ] відображення прогресу (кільце, % у списку) |
| `timesCorrect` | Int | [НОВЕ] скільки разів «Знаю» за весь час |
| `timesWrong` | Int | [НОВЕ] скільки разів «Не знаю» |
| `boxLevel` | Int 0–5 | [НОВЕ] Leitner-бокс; 0 = нове (не треноване), 5 = вивчене |
| `lastReviewedAtEpochMillis` | Long? | [НОВЕ] час **останнього тренування** (окремо від `updatedAt`!) |
| `dueAtEpochMillis` | Long? | [НОВЕ] коли слово знову «дозріє» = `lastReviewed + interval(boxLevel)` |

> `lastReviewedAt`/`dueAt` оновлюються **лише тренуванням**, не редагуванням слова — це і є чесна «давність практики».

### Сервер — нові колонки `topic_words` (міграція 0008, ескіз)

```sql
ALTER TABLE topic_words
  ADD COLUMN IF NOT EXISTS times_correct      integer NOT NULL DEFAULT 0,
  ADD COLUMN IF NOT EXISTS times_wrong        integer NOT NULL DEFAULT 0,
  ADD COLUMN IF NOT EXISTS box_level          integer NOT NULL DEFAULT 0,
  ADD COLUMN IF NOT EXISTS last_reviewed_at   timestamptz,
  ADD COLUMN IF NOT EXISTS due_at             timestamptz;
-- box_level у діапазоні 0..5, перевірка CHECK як у knowledge_percent
```

Ці поля додаються у `TopicWordResponseDto`/`ClientTopicWordSyncDto` (див. doc 06) і їдуть через `applySync`/`syncTopics`.

## 4. [НОВЕ] Алгоритм відбору сесії (10 слів)

### 4.1 Категорії слів пулу (зі словників, обраних на setup-екрані)

| Категорія | Умова |
|---|---|
| **Нове** | `boxLevel == 0` (ще не тренувалось) |
| **Дозріле (due)** | `boxLevel ≥ 1` і `dueAt ≤ now` |
| **На кулдауні** | `lastReviewedAt` у межах кулдаун-вікна (напр. показане за останні 8 год / останню сесію) → **тимчасово виключається** |
| **Майбутнє** | `boxLevel ≥ 1` і `dueAt > now` (ще «спить») |
| **Вивчене** | `boxLevel == 5` і `knowledgePercent ≥ 90` → у пул потрапляє рідко |

### 4.2 Квоти сесії (конфігурабельні)

Ціль — `SESSION_SIZE = 10`: `≈ 7` повторення + `≈ 3` нових.

```
newQuota    = min(NEW_PER_SESSION = 3, доступні «Нові»)
reviewQuota = SESSION_SIZE - newQuota
```

Адаптація, якщо не вистачає:
- Мало «Нових» → решту добираємо з повторення.
- Мало «Дозрілих» → добираємо найслабші з «Майбутнього» (не чекаємо штучно), потім — рідко — з «Вивчених».
- Усього < 10 слів у словниках → сесія = скільки є (див. крайові випадки §9).

### 4.3 Пріоритет-бал слова (для зваженого рандому в пулі повторення)

```
weight(word) =
      weakness(word)        // 1 + (100 - knowledgePercent) / 50      → слабші важать більше
    * overdue(word)         // 1 + clamp((now - dueAt) / interval, 0, 3) → прострочені важать більше
    * errorRate(word)       // 1 + timesWrong / (timesCorrect + timesWrong + 1) → помилкові важать більше
    * cooldown(word)        // 0, якщо на кулдауні; інакше 1
```

Відбір повторення — **зважений рандом без повторів** (weighted sampling) за `weight`, поки не набереться `reviewQuota`. Нові — рівномірний рандом серед «Нових». Фінальний порядок карток — додатково перемішується (`shuffle`), щоб слабкі не йшли купкою.

### 4.4 Псевдокод

```
fun buildSession(words: List<Word>, now): List<Word> {
    val active   = words.filterNot { onCooldown(it, now) }
    val news     = active.filter { it.boxLevel == 0 }
    val due      = active.filter { it.boxLevel >= 1 && it.dueAt <= now }
    val future   = active.filter { it.boxLevel >= 1 && it.dueAt > now }

    val pickedNew    = news.shuffled().take(min(NEW_PER_SESSION, news.size))
    val reviewQuota  = SESSION_SIZE - pickedNew.size

    val reviewPool   = (due + future.sortedBy { it.knowledgePercent })   // due пріоритетніші
    val pickedReview = weightedSampleWithoutReplacement(reviewPool, reviewQuota) { weight(it, now) }

    return (pickedNew + pickedReview).shuffled().take(SESSION_SIZE)
}
```

> Якщо після виключення кулдауну слів < 10 — кулдаун **тимчасово послаблюється** (беремо й нещодавно показані), щоб сесія не була порожньою, але вони лишаються з найнижчою вагою.

## 5. [НОВЕ] Leitner-інтервали (дозрівання)

`interval(boxLevel)` — коли слово знову стане «дозрілим». Усі значення **конфігурабельні**.

| boxLevel | Стан | Інтервал до `dueAt` | knowledge% (орієнтир) |
|---|---|---|---|
| 0 | нове | — (одразу в пулі) | 0 |
| 1 | щойно почали | ~10 хв (та сама/наступна сесія) | ~20 |
| 2 | | 1 день | ~40 |
| 3 | | 3 дні | ~60 |
| 4 | | 7 днів | ~80 |
| 5 | вивчене | 16 днів (далі рідко) | 90–100 |

## 6. [НОВЕ] Обробка відповіді (бінарна)

### «Знаю» (correct)
```
timesCorrect += 1
boxLevel       = min(5, boxLevel + 1)
knowledgePercent = min(100, knowledgePercent + KNOW_DELTA)     // напр. +12
lastReviewedAt = now
dueAt          = now + interval(boxLevel)
```

### «Не знаю» (wrong)
```
timesWrong  += 1
boxLevel       = 1                                            // падіння в бокс 1 (не в 0 — слово вже відоме системі)
knowledgePercent = max(0, knowledgePercent - MISS_DELTA)      // напр. -20 (помилка коштує більше)
lastReviewedAt = now
dueAt          = now + interval(1)                            // скоро знову
// + слово ПЕРЕЧЕРГОВУЄТЬСЯ в кінець поточної колоди (показуємо ще раз цієї ж сесії — кнопка «Далі»)
```

> `MISS_DELTA > KNOW_DELTA` навмисно: одна помилка відкочує більше, ніж одна вдала відповідь додає — слабкі слова не «вистрибують» у вивчені за один клік.
> `knowledgePercent` — згладжений показник для UI; `boxLevel`+`dueAt` — двигун розкладу. Тримаємо обидва узгоджено.

## 7. [НОВЕ] Флов екранів

```
PracticeSetup (вибір тем, мультивибір)
      │  кнопка «Почати»
      ▼
buildSession() → колода з 10 карток
      │
      ▼
PracticeFlipCard (для кожної картки)
   tap → переворот (показ перекладу + контекст)
   «Не знаю» → wrong → перечергувати в кінець
   «Знаю»    → correct → наступна
      │  колода вичерпана
      ▼
PracticeDone (статистика сесії)
   «N з 10 правильно · +X% знань»
   «Ще раз» → нова buildSession() (новий seed + кулдаун → ІНШІ слова)
   «До словників»
```

Стани: `setup` (нема обраних тем → кнопка неактивна), `empty` (нема слів узагалі), `running` (картки), `done` (підсумок).

## 8. [НОВЕ] Чому не «ті самі 10 слів» — явні механізми

1. **Кулдаун** — слово з минулої/поточної сесії виключається на `COOLDOWN_WINDOW` (напр. 8 год) → не повертається одразу.
2. **Зважений рандом** — навіть однаковий пул дає щоразу інший набір (на відміну від детермінованого `sortedBy`).
3. **Нові слова щосесії** — до 3 ще не тренованих → завжди є «свіже».
4. **Leitner-дозрівання** — вивчені слова йдуть у довгі інтервали й випадають із щоденного пулу.
5. **«Ще раз»** бере **новий seed** і враховує оновлений кулдаун → друга сесія поспіль дає інші картки.

## 9. [НОВЕ] Крайові випадки

| Випадок | Поведінка |
|---|---|
| У словниках < 10 слів | Сесія = усі доступні (напр. 4 картки). |
| Усі слова «вивчені» (бокс 5, не дозріли) | Показуємо стан «Все повторено 🎉 — повертайся пізніше», пропонуємо «Тренувати все одно» (ігнорує `dueAt`). |
| Жодного нового слова | Сесія = лише повторення (newQuota=0). |
| Обрано кілька тем | Пул об'єднується; квоти й ваги — глобально по пулу. |
| Офлайн | Тренування працює локально; зміни полів синкаються при появі мережі (D1). |
| Брак не-кулдаун-слів | Кулдаун тимчасово послаблюється (див. §4.4). |

## 12. [НОВЕ] Тренування «Слово в контексті» (design board 6, гілка `feature/context-practice`)

Другий вид тренування для слів із **2+ збереженими перекладами**. Одиниця навчання —
**пара** «слово + конкретний переклад»; пара = наявний `WordEntry` (той самий `source`,
різні `translation`), прогрес пари = його ж `knowledgePercent` (±20, як у класиці).
Контекст пари — перше речення з `details.senses[].examples`; пари без прикладів у
тренування не потрапляють (лишаються дистракторами).

**Формати (подає система за рівнем пари):**
- `<40%` — **впізнавання**: EN-речення з жовтою підсвіткою слова + 2–4 чипи-варіанти;
  дистрактори — інші переклади цього ж слова. Помилка → міні-підказка з реченням
  сплутаної пари + «Далі»; правильно → зелений стан і авто-перехід ~0.7с.
- `40–59%` — **згадування**: речення + «Як перекладається тут?», тап = розворот,
  «Знаю / Не знаю» як у класиці; на звороті інші значення слова димчасто.
- `≥60%` — **вживання (UK→EN)**: EN-речення з пропуском `____` + підказка-переклад,
  варіанти — EN-слова юзера з інших груп. *(Відхилення від борду: у борді UK-речення;
  у нас перекладених речень немає, тому cloze це [МАЙБУТНЄ] після генерації перекладів.)*
- **Метчинг** («фінал слова»): коли в слова ≥3 пари з реченнями і всі `≥50%` — одна
  картка з colour-coded з'єднанням речень і перекладів; бали — за першу спробу кожної пари.

**Сесія:** до 10 карток, найслабші пари першими, анти-повтор одного source поспіль,
максимум 1 метчинг за раунд. Сетап: перемикач «Класика / Слово в контексті»
(`PracticeModeCards`), рядки словників показують «N пар у контексті», футер — суму пар.
Empty-стан із CTA «До словників». Результат: кільце % + блок **«Найплутаніша пара»**
(лог помилок впізнавання/метчингу, найчастіша плутанина).

Код ізольовано в `presentation/ContextPractice.kt`; в `App.kt` — лише перемикач режиму
в сетапі та маршрутизація. Відкат = revert гілки або дефолт `PracticeMode.Classic`.

**[МАЙБУТНЄ]:** переклади речень для повного UK→EN; бекфіл прикладів через gateway для
пар без речень; «помилки раунду першими в наступному» (зараз — лише сортування за
knowledge); квік-дод незнайомого слова з речення (окремий дизайн, board 7).
