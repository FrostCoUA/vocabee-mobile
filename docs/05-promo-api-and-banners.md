# 05 — Config-Driven Promo API та банери

> Розділ описує **нову** підсистему промо-кампаній і банерів. Усе, що тут описано, позначено **[НОВЕ]** — у поточному коді промо немає взагалі. База, яку ми розширюємо, — наявні банери в `App.kt` (`BeeWalletBanner`, `AnonymousFreeLimitBanner`, `CriticalBeeBanner`, `BeeRewardSheet`, `AuthRequiredSheet`).
>
> Джерело істини рішень: D2 (анонім без монеток), D4 (config-driven promo API). Економічні константи: `wallet.constants.ts`, `VocabeeStore.kt`.

---

## 1. Концепція **[НОВЕ]**

| Принцип | Опис |
|---|---|
| Бекенд віддає промо | `GET /v1/promos` повертає список **активних** промо з уже порахованим прогресом конкретного юзера. |
| Клієнт лише рендерить | Клієнт **не містить промо-логіки**. Він бере конфіг і малює банер + ботомшит за полями `banner` і `sheet`. Жодних захардкоджених сум, лічильників, умов скидання чи назв кампаній на клієнті. |
| Базова винагорода лишається | Перегляд реклами завжди дає базові **+10** монеток (`REWARDED_AD_BEE_AMOUNT = 10`). Промо — це **бонус зверху** (наприклад «10 реклам → ще +20»). База і промо нараховуються окремими сутностями на сервері. |
| Анонім промо не бачить (D2) | Анонімний користувач **не має монеток, реклами й промо**. Замість промо-банера йому показується банер гостьового режиму (`AnonymousFreeLimitBanner`). Промо вмикається **тільки** після входу через Google. |
| Сервер авторитетний (D1) | Прогрес промо, факт видачі призу (`claim`) і нарахування за рекламу рахує/верифікує сервер. Клієнт може показати оптимістичне просування лічильника, але звіряє його з відповіддю сервера. |

**Чому config-driven.** Власник продукту може запустити/зупинити/змінити кампанію (суми, тексти, іконки, пороги, вікна скидання) **без релізу застосунку** — змінюється лише серверний конфіг. Клієнт залишається «тупим» рендерером.

**Покликання на наявний код (база для розширення):**

| Компонент | Розташування | Роль після розширення |
|---|---|---|
| `BeeWalletBanner` | `App.kt:1562` | Фолбек-банер для авторизованого без активних промо (баланс + «відео +10»). |
| `AnonymousFreeLimitBanner` | `App.kt:1502` | Банер аноніма (гостьовий режим), промо для нього не показуємо. |
| `CriticalBeeBanner` | `App.kt:1611` | Накладається поверх, коли баланс `≤ CriticalBeeThreshold (3)`. |
| `BeeRewardSheet` | `App.kt:1067` | База для **PromoSheet** (структура ботомшита). |
| `AuthRequiredSheet` | `App.kt:1144` | База для `locked`-стану промо (кнопка `sign_in`). |
| `BeeBalanceBadge` | `App.kt:1642` | Перевикористовуємо для бейджа балансу/винагороди. |

## 2. JSON-схема конфіга **[НОВЕ]**

Кореневий обʼєкт відповіді `GET /v1/promos`:

```jsonc
PromoConfig {
  "promos": Promo[],   // активні промо для цього юзера, відсортовані сервером за priority desc
  "serverTime": string // ISO-8601 UTC, для розрахунку зворотного відліку до resetsAt на клієнті
}
```

### Promo

| Поле | Тип | Опис |
|---|---|---|
| `id` | `string` | Стабільний ідентифікатор кампанії (`milestone_ads_v1`, `daily_streak_v1`, …). Використовується в `POST /v1/promos/{id}/claim`. |
| `type` | enum | `"milestone_ads" \| "daily_streak" \| "one_time" \| "leaderboard"`. Підказка клієнту, **яку візуалізацію прогресу** малювати (бар / точки днів / кільце). Логіки клієнт із цього не виводить. |
| `state` | enum | `"active" \| "claimable" \| "completed" \| "locked" \| "expired"`. Визначає вигляд ботомшита й набір активних кнопок (див. §5). |
| `priority` | `number` | Вищий = вище в черзі банерів. Сервер уже сортує, але клієнт теж може покладатися на це поле. |
| `reward` | `{ bees: number }` | **Конфігурабельна** винагорода за виконання промо. Окрема від базових +10 за рекламу. |
| `progress` | `Progress` | Прогрес юзера (див. нижче). |
| `resetsAt` | `string \| null` | ISO-8601 момент скидання прогресу/вікна (для `daily_streak`, `leaderboard`, повторюваного `milestone`). `null` — без скидання (`one_time`). |
| `banner` | `Banner` | Копірайт і вигляд **банера** на головному екрані. |
| `sheet` | `Sheet` | Вміст **ботомшита**, що відкривається з банера. |

### Progress

| Поле | Тип | Опис |
|---|---|---|
| `current` | `number` | Поточне значення основного лічильника (напр. реклам усього / днів стріку). |
| `target` | `number` | Ціль основного лічильника. |
| `unit` | `string` | Одиниця основного лічильника для копірайту: `"ads"`, `"days"`, `"signups"`, `"rank"`. |
| `todayCurrent` | `number?` | Опційно: сьогоднішнє значення (для `daily_streak` — скільки реклам сьогодні). |
| `todayTarget` | `number?` | Опційно: денна ціль (для `daily_streak` — мінімум реклам/день). |

### Banner

| Поле | Тип | Опис |
|---|---|---|
| `icon` | `string` | Ключ іконки з реєстру клієнта (`gift`, `flame`, `trophy`, `sparkle`, …). Невідомий ключ → дефолт `sparkle`. |
| `accent` | enum | `"purple" \| "orange" \| "yellow" \| "green"`. Мапиться на `PrototypeColor`. |
| `title` | `string` | Заголовок банера (українською, прийшов із сервера). |
| `subtitle` | `string` | Підзаголовок/прогрес-копірайт банера. |

### Sheet

| Поле | Тип | Опис |
|---|---|---|
| `title` | `string` | Заголовок ботомшита. |
| `body` | `string` | Пояснення кампанії (1–3 речення). |
| `progressView` | `ProgressView` | Що і як показати на візуалізації прогресу. |
| `buttons` | `Button[]` | Список CTA-кнопок (рендеряться згори вниз). |

### ProgressView

| Поле | Тип | Опис |
|---|---|---|
| `current` | `number` | Значення основного прогрес-індикатора. |
| `target` | `number` | Ціль основного прогрес-індикатора. |
| `unit` | `string` | Одиниця (`"ads"`, `"days"`…). |
| `secondary` | `{ current, target, unit, label }?` | Опційний другий індикатор (для `daily_streak`: «сьогодні 4 з 3 реклам»). |

### Button

| Поле | Тип | Опис |
|---|---|---|
| `kind` | enum | `"watch_ad" \| "claim_reward" \| "open_deeplink" \| "sign_in" \| "dismiss"`. Визначає **дію** клієнта. |
| `label` | `string` | Текст кнопки (із сервера). |
| `style` | enum | `"primary" \| "secondary" \| "ghost"`. Візуальний стиль. |
| `payload` | `object?` | Дані для дії: для `open_deeplink` — `{ "route": "leaderboard" }`; для `claim_reward` зазвичай порожнє (id у URL). |

**Дії кнопок (клієнт):**

| `kind` | Дія клієнта |
|---|---|
| `watch_ad` | Запускає rewarded ad → після завершення `POST /v1/wallet/rewarded-ad` (база +10 + просування ad-лічильників). У відповіді приходять оновлені промо+баланс. |
| `claim_reward` | `POST /v1/promos/{id}/claim` (ідемпотентний). Сервер перевіряє прогрес і нараховує `reward.bees`. |
| `open_deeplink` | Навігація за `payload.route` (напр. екран лідерборда). |
| `sign_in` | Запускає Google-вхід (для `locked` у аноніма). |
| `dismiss` | Закриває ботомшит. |

## 4. Банер — стани й пріоритет показу **[НОВЕ]**

На головному екрані (`DictionariesHomeScreen`, `App.kt:1354`) у стрічці є **один** банерний слот (`item(span = maxLineSpan)`, `App.kt:1386–1401`). Розширюємо логіку вибору банера так:

### Пріоритет вибору банера

1. **Анонім (D2)** → завжди `AnonymousFreeLimitBanner` (гостьовий режим). Промо ігноруються.
2. **Авторизований + є активні промо** → **PromoBanner** з конфіга з найвищим `priority`. Якщо активних промо кілька — показуємо найвищий `priority`, опційно **карусель** (горизонтальний пейджер по промо, відсортованих за `priority desc`).
3. **Авторизований без активних промо** → `BeeWalletBanner` (баланс + «дивись відео +10»). Фолбек.
4. **Критичний баланс `≤ CriticalBeeThreshold (3)`** → `CriticalBeeBanner` показується **поверх / замість** основного банера (найвищий пріоритет уваги), бо це блокер для пошуку перекладів. Якщо одночасно є `claimable`-промо, можна показати критичний банер, а `claimable` лишити крапкою/бейджем — **Припущення (уточнити)** з дизайном.

> **Карусель vs один банер** — **Припущення (уточнити):** для MVP достатньо показати один найпріоритетніший промо-банер; карусель — як ітерація 2. У документі залишаємо обидва варіанти, фінал за дизайном.

### Мапінг `accent` → колір

| `accent` | `PrototypeColor` | Фон банера |
|---|---|---|
| `purple` | `PrototypeColor.Purple` | `PrototypeColor.Tint` |
| `orange` | `PrototypeColor.Orange` | `PrototypeColor.NotePeach` |

## 5. Промо-ботомшит — структура **[НОВЕ]**

Базується на `BeeRewardSheet` (`App.kt:1067`) через спільний `PrototypeBottomSheet`. Новий `PromoSheet(promo: Promo)` рендериться **повністю** з `promo.sheet`.

### Структура зверху вниз

1. **Бейдж винагороди** — `BeeBalanceBadge`-стиль (`App.kt:1642`) з `reward.bees` і `Sparkle`-іконкою. Колір — за `banner.accent`.
2. **Заголовок** — `sheet.title`.
3. **Пояснення** — `sheet.body`.
4. **Візуалізація прогресу** — за `sheet.progressView` і `promo.type`:
   - `milestone_ads` → горизонтальний прогрес-бар `current/target` + копірайт «4 з 10».
   - `daily_streak` → **5 точок днів** (заповнені = пройдені дні), плюс `secondary` як міні-бар «сьогодні 4 з 3 реклам».
   - `one_time` → проста галочка/стан «готово до отримання».
   - `leaderboard` → **кільце** або текст «Ваше місце: 14 · поріг топ-10» + зворотний відлік до `resetsAt`.
5. **Кнопки** — `sheet.buttons` згори вниз, стиль за `style` (`primary` → `PrototypeColor.Purple`, `secondary` → `NeutralSurface`, `ghost` → текстова).

### Стани ботомшита (`promo.state`)

| `state` | Вигляд | Активні кнопки | Поведінка |
|---|---|---|---|
| `active` | прогрес-бар/точки в нейтральному акценті | `watch_ad` (primary), `dismiss` | `watch_ad` → rewarded ad → +10 база + просування лічильника; після відповіді сервера прогрес оновлюється, можливий перехід у `claimable`. |
| `claimable` | прогрес заповнений, **підсвітка** (accent `green`), бейдж пульсує | `claim_reward` (primary), `dismiss` | `claim_reward` → `POST /v1/promos/{id}/claim`; сервер нараховує `reward.bees`; для повторюваних — прогрес обнуляється, промо повертається в `active`. |
| `completed` | «Виконано», прогрес 100%, без CTA-екшну | лише `dismiss` (+ опц. таймер до `resetsAt`) | Нічого не нараховує. Для повторюваних показує «оновиться через …» до `resetsAt`. |
| `locked` | заблокований вигляд, пояснення про вхід | `sign_in` (primary), `dismiss` | Для аноніма / промо, що вимагає акаунта. `sign_in` запускає Google-вхід (як `AuthRequiredSheet`, `App.kt:1144`). |
| `expired` | приглушений, «кампанія завершилась» | лише `dismiss` | Не повинно потрапляти в активну стрічку; сервер зазвичай фільтрує. |

> **Звʼязок `watch_ad` ↔ база +10.** Кнопка `watch_ad` у будь-якому промо-шиті використовує **той самий** rewarded-ad флоу, що й `BeeRewardSheet`. База +10 нараховується завжди; промо-лічильники просуваються тим же запитом `POST /v1/wallet/rewarded-ad`. Подвійних нарахувань немає — і база, і просування промо — одна серверна транзакція.

## 6. Лідерборд топ-10 **[НОВЕ]**

Окремий екран/шторка `LeaderboardScreen`, відкривається кнопкою `open_deeplink` (`payload.route = "leaderboard"`) з промо-шита `leaderboard`.

### Вміст екрана

| Блок | Опис |
|---|---|
| Заголовок | «Топ-10 тижня за переглядами реклами» + зворотний відлік до `resetsAt` (наступний понеділок). |
| Список топ-10 | Рядки: позиція, аватар/імʼя (або анонімізований нік), кількість переглядів за тиждень. Перші 10 підсвічені — вони отримають `reward.bees` (+50, конфігурабельно). |
| «Моє місце» | Закріплений рядок «Ви: 14 місце · 23 перегляди · до топ-10 ще N». Якщо юзер у топ-10 — підсвічений у списку. |
| Винагорода | Бейдж «+50 монеток для топ-10», скидання щопонеділка. |

### Джерело даних

- **Нова таблиця/агрегат на сервері:** `ad_watch_weekly` (агрегат переглядів за поточний ISO-тиждень на користувача), або матеріалізована вʼюшка поверх лога подій реклами.
- **Скидання щотижня:** нарахування топ-10 і обнулення тижневого агрегату виконує плановий джоб щопонеділка 00:00 UTC (cron). Нараховані +50 ідуть через ту саму ідемпотентну механіку, що й `claim` (одна видача на тиждень на юзера).
- Ендпоінт: `GET /v1/promos/leaderboard/ad-watchers` (див. §7).

## 7. Ендпоінти — пропонований контракт **[НОВЕ]**

Усі під `@UseGuards(JwtAccessGuard)` (як наявний `WalletController`, `wallet.controller.ts:13`). Анонім без токена → промо-ендпоінти повертають `promos: []` / `401`.

### `GET /v1/promos`

Список активних промо з прогресом конкретного юзера.

```jsonc
// 200 OK
{
  "serverTime": "2026-06-25T09:00:00Z",
  "promos": [ /* Promo[], відсортовані за priority desc */ ]
}
```

### `POST /v1/promos/{id}/claim`

Ідемпотентна видача призу. Сервер **сам** перевіряє прогрес — не вірить клієнту.

```jsonc
// Тіло запиту не обовʼязкове; id у URL.
// 200 OK
{
  "claimed": true,
  "reward": { "bees": 20 },
  "balance": 84,            // оновлений баланс монеток
  "promos": [ /* оновлений список промо, напр. milestone знову active з current=0 */ ]
}
// 409 Conflict — прогрес ще не виконано (state != claimable)
// Повторний claim того ж id у тому ж вікні → 200 з claimed:false (ідемпотентно, без подвійної видачі)
```

Ідемпотентність: ключ `(userId, promoId, resetWindow)`; повторний виклик у тому ж вікні скидання не нараховує вдруге.

### `POST /v1/wallet/rewarded-ad`

Розширення наявного ендпоінта (`wallet.controller.ts:25`). Зараз — «голий POST» без верифікації, нараховує лише `REWARDED_AD_BEE_AMOUNT`. **[НОВЕ]** додаємо:

- **Серверну верифікацію** (Server-Side Verification / разовий `nonce`) — щоб неможливо було накрутити нарахування підробленим запитом.
- **Просування ad-лічильників** (всього/сьогодні/стрік) у тій самій транзакції.
- У відповідь — оновлений баланс **і** оновлені промо.

```jsonc
// Запит
{ "nonce": "ssv-one-time-token-from-ad-sdk" }
// 200 OK
{
  "balance": 64,                 // після базових +10
  "credited": { "bees": 10 },    // база
  "promos": [ /* оновлені промо: лічильники зросли, можливий перехід active→claimable */ ]
}
// 400 — невалідний/повторно використаний nonce (нічого не нараховуємо)
```

### `GET /v1/promos/leaderboard/ad-watchers`

```jsonc
// 200 OK
{
  "resetsAt": "2026-06-29T00:00:00Z",
  "reward": { "bees": 50 },
  "top": [
    { "rank": 1, "displayName": "Олена К.", "adsThisWeek": 41, "isMe": false },
    { "rank": 2, "displayName": "Ви",        "adsThisWeek": 38, "isMe": true  }
    // ... до 10
  ],
  "me": { "rank": 14, "adsThisWeek": 23, "isMe": true }  // присутній, навіть якщо поза топ-10
}
```

| Метод | Шлях | Призначення | Ідемпотентність |
|---|---|---|---|
| `GET` | `/v1/promos` | Активні промо + прогрес юзера | — (read) |
| `POST` | `/v1/promos/{id}/claim` | Видача призу, серверна перевірка прогресу | Так, ключ `(userId, promoId, resetWindow)` |
| `POST` | `/v1/wallet/rewarded-ad` | База +10 (верифіковано) + просування ad-лічильників | Так, разовий `nonce` |
| `GET` | `/v1/promos/leaderboard/ad-watchers` | Топ-10 тижня + «моє місце» | — (read) |

## 8. Серверні лічильники для промо **[НОВЕ]** — нова бекенд-підсистема

Промо неможливі без серверного обліку переглядів реклами і станів видачі. Це **нова підсистема** поряд із `wallet` (модуль на кшталт `promo`).

### Лічильники реклами (на користувача)

| Поле | Тип | Призначення |
|---|---|---|
| `adsTotal` | `int` | Усього переглядів реклами за весь час. |
| `adsToday` | `int` | Перегляди за поточну добу (UTC); обнуляється на зміну дня. |
| `lastAdWatchDate` | `date` | Дата останнього перегляду (для денного обнулення й стріку). |
| `streakDays` | `int` | Поточна довжина стріку днів (для `daily_streak`). |
| `lastStreakDate` | `date` | Дата останнього дня, що зарахувався до стріку. |
| `adsThisWeek` | `int` | Перегляди за поточний ISO-тиждень (для лідерборда). |
| `weekKey` | `string` | Ключ тижня (`2026-W26`); зміна ключа → обнулення `adsThisWeek`. |

### Стан видачі промо (на користувача × промо)

| Поле | Тип | Призначення |
|---|---|---|
| `promoId` | `string` | Кампанія. |
| `progressCurrent` | `int` | Поточний прогрес (для повторюваних — від останнього скидання). |
| `claimedAt` | `timestamp \| null` | Коли востаннє забрано приз (для ідемпотентності та `one_time`). |
| `resetWindow` | `string` | Ідентифікатор вікна скидання (`2026-W26`, дата дня, `lifetime`) — частина ключа ідемпотентності. |

### Похідні сутності

- **Тижневий агрегат для лідерборда** (`ad_watch_weekly`): `(userId, weekKey) → adsThisWeek`, з індексом для топ-10 і пошуку «мого місця».
- **Лог переглядів / nonce-стор**: разові `nonce` від реклами, щоб гарантувати ідемпотентність `rewarded-ad` (захист від накрутки/повторів).

### Правила обнулення

| Сутність | Коли скидається |
|---|---|
| `adsToday` | На зміну доби (UTC) — лінива перевірка `lastAdWatchDate` при записі. |
| `streakDays` | Якщо пропущено день із невиконаною денною нормою → стрік → 0. |
| `adsThisWeek` / лідерборд | Щопонеділка 00:00 UTC (cron-джоб + нарахування топ-10). |
| `milestone` прогрес | Після успішного `claim` (повторювана кампанія). |
| `one_time` | Ніколи (фіксується `claimedAt`). |

> **Звʼязок з D1 (сервер авторитетний).** Усі лічильники й видачі живуть **тільки** на сервері. Клієнт показує оптимістичне просування для UX, але остаточні значення завжди беруться з відповідей `GET /v1/promos`, `POST /v1/wallet/rewarded-ad`, `POST /v1/promos/{id}/claim`.
