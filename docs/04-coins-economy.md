# 04. Економіка монеток (beecoins)

> «Монетки» = **beecoins** (бджолина тематика, іконка Sparkle / стільники).
> Для кожної поведінки явно позначено: **[ЗАРАЗ]** — як працює в поточному коді, **[НОВЕ]** — затверджена зміна.
> Джерела істини констант: `wallet.constants.ts`, `VocabeeStore.kt:23-29`.

---

## 0. Константи економіки

| Константа | Значення | Сервер (`wallet.constants.ts`) | Клієнт (`VocabeeStore.kt`) |
|---|---|---|---|
| Стартовий баланс | **50** | `INITIAL_BEE_BALANCE`:1 | `InitialBeeBalance`:23 |
| Винагорода за рекламу (база) | **10** | `REWARDED_AD_BEE_AMOUNT`:2 | `RewardBeeAmount`:24 |
| Безкоштовний ліміт словників | **2** | `FREE_DICTIONARY_LIMIT`:3 | `FreeDictionaryLimit`:25 |
| Вартість словника понад ліміт | **10** | `DICTIONARY_CREATION_BEE_COST`:4 | `DictionaryCreationBeeCost`:27 |
| Вартість пошуку перекладу | **1** | `TRANSLATION_SEARCH_BEE_COST`:5 | `TranslationSearchBeeCost`:28 |
| Ліміт слів для аноніма | **50** | — (немає, лише клієнт) | `AnonymousFreeWordLimit`:26 |

## 1. Звідки беруться монетки і на що витрачаються

### Надходження (income)

| Джерело | Сума | Стан | Позначка |
|---|---|---|---|
| Стартовий баланс при створенні акаунта | **+50** | authenticated | [ЗАРАЗ] |
| Перегляд реклами (база) | **+10** | authenticated | [ЗАРАЗ] |
| Промо-бонуси (milestone / daily_streak / one_time / leaderboard) | конфігурабельні (напр. +20 / +50 / +50 / +50) | authenticated | **[НОВЕ]** — деталі у файлі **05-promo** |

- **Стартові 50 [ЗАРАЗ].** На клієнті дефолт `beeBalance = InitialBeeBalance` у `VocabeeState` (`VocabeeStore.kt:41`) і відновлюється з prefs у `initialState()` (`VocabeeStore.kt:266`). На сервері — `INITIAL_BEE_BALANCE` нараховується при реєстрації користувача (значення в колонці `users.beeBalance`).
- **+10 за рекламу (база) [ЗАРАЗ].** Сервер: `POST /wallet/rewarded-ad` → `addBees(userId, REWARDED_AD_BEE_AMOUNT)` (`wallet.controller.ts:28-31`). Клієнт: подія `VocabeeEvent.AddBees(amount = RewardBeeAmount)` → `addBees()` (`VocabeeStore.kt:354-360`).
- **Промо-бонуси [НОВЕ].** За **D4** — config-driven, лишаються **зверху** базових +10. Клієнт лише рендерить із конфіга; жодної захардкодженої промо-логіки. Деталі — файл **05-promo**.

### Витрати (spend)

| Дія | Сума | Стан | Позначка |
|---|---|---|---|
| Створення словника понад 2 безкоштовні (3-й+) | **−10** | authenticated | [ЗАРАЗ] |
| Пошук перекладу (1 запит) | **−1** | authenticated | [ЗАРАЗ] (клієнтський гейт; **[НОВЕ]** — серверне списання) |

- **Видалення словника НЕ повертає монетки** (за **D3**), але показуємо Undo-снекбар (~5–10 с). Деталі видалення — файл 03 / 05.

## 3. Анонім vs Авторизований (D2)

### Принцип D2 — АНОНІМ БЕЗ МОНЕТОК
Анонімні користувачі **не мають** монеток, реклами й промо. У них безкоштовний пробний режим: **максимум 2 словники, максимум 50 слів усього**. Монетки / реклама / промо вмикаються **тільки** після входу через Google. Реєстрація — момент апгрейду.

| Властивість | Анонім | Авторизований |
|---|---|---|
| Монетки | **немає** (early-return) | є, баланс на сервері |
| Ліміт словників | жорсткий: **2** | м'який: 2 безкоштовні, далі −10/шт |
| Ліміт слів | жорсткий: **50 усього** | без ліміту |
| Реклама / промо | **недоступні** | доступні |
| Джерело балансу | — | сервер (авторитетний, D1) |

### Реалізація early-return [ЗАРАЗ]
`addBees`/`spendBees` роблять ранній вихід, якщо `!isAuthenticated()`:

- `addBees(amount)` — `VocabeeStore.kt:354-360`: `if (!isAuthenticated()) return` → аноніму монетки **не нараховуються**.
- `spendBees(amount)` — `VocabeeStore.kt:362-370`: `if (!isAuthenticated()) return false` → у аноніма списання завжди **провалюється** (бо в нього немає економіки).

### Гейти доступу [ЗАРАЗ]

**`canCreateTopic()`** — `VocabeeStore.kt:221-227`
```
authenticated → topics.size < 2  АБО  beeBalance >= 10
anonymous     → topics.size < 2            (жорсткий ліміт, монетки не враховуються)
```
- Анонім: дозволено лише доки словників < 2.
- Авторизований: перші 2 безкоштовні; 3-й+ — лише якщо вистачає 10 монеток.

**`canSearchTranslation()`** — `VocabeeStore.kt:229-235`
```
authenticated → beeBalance >= 1
anonymous     → !anonymousWordLimitReached()   (тобто < 50 слів усього)
```
- Анонім: дозволено, доки не вперся в ліміт 50 слів (монетки не при чому).
- Авторизований: дозволено, доки баланс ≥ 1.

**`canAddWordToDictionary()`** — `VocabeeStore.kt:237-239`
```
authenticated → true (без ліміту)
anonymous     → !anonymousWordLimitReached()   (< 50 слів усього)
```

**Допоміжні предикати анонімних лімітів:**
- `anonymousDictionaryLimitReached()` — `VocabeeStore.kt:213-215`: `!isAuthenticated() && topics.size >= 2`.
- `anonymousWordLimitReached()` — `VocabeeStore.kt:217-219`: `!isAuthenticated() && totalWordCount() >= 50`.

**Списання при пошуку:** `spendTranslationBee()` — `VocabeeStore.kt:241-244`:
```
anonymous     → повертає canSearchTranslation() (нічого не списує — у аноніма немає монеток)
authenticated → spendBees(1)
```

> **[НОВЕ за D1]:** ці клієнтські гейти **лишаються** як оптимістичний UX-шар, але остаточну квоту/списання вирішує сервер (`/search`, `applySync`, `create`). Клієнт після відповіді виставляє підтверджений баланс через `SetBeeBalance`.

## 4. [ЗАРАЗ] Баг подвійного списання за створення словника → [НОВЕ] фікс (D1)

### [ЗАРАЗ] Проблема
За створення платного словника (3-й+) монетки знімаються **двічі**:

1. **Клієнт:** `createTopic()` → `if (shouldCharge && !spendBees(DictionaryCreationBeeCost)) return` (`VocabeeStore.kt:277-278`) — мінус 10 локально.
2. **Сервер:** `TopicsService.create()` → `if (count >= FREE_DICTIONARY_LIMIT) await walletService.spendBees(userId, DICTIONARY_CREATION_BEE_COST)` (`topics.service.ts:104-106`) — мінус 10 на сервері.

> Результат: фактично **−20** за один словник (−10 локально оптимістично + −10 на сервері), і при наступній звірці клієнт побачить розсинхрон. Те саме стосується шляху `applySync` — наразі `applyClientSync` (`topics.service.ts:318-333`) взагалі **не списує** монетки за залиті словники, тобто квота не enforced на синхронізації.

### [НОВЕ за D1] Фікс — сервер авторитетний, клієнт оптимістичний + звірка
- **Сервер — єдине джерело істини** для балансу й усіх списань. Списання за створення словника лишається **тільки** на сервері (`topics.service.ts` `create` + **додати** enforcement у `applyClientSync`).
- **Клієнт** списує **оптимістично** (миттєвий UX), але **не вважає** локальний баланс достеменним — після відповіді сервера виставляє підтверджений баланс через `VocabeeEvent.SetBeeBalance` → `setBeeBalance()` (`VocabeeStore.kt:372-376`).
- **`applySync` МУСИТЬ** валідувати квоти й списувати монетки, відхиляючи/обрізаючи понадлімітне (зараз не робить — це треба додати).

### Рекомендований контракт (клієнт ↔ сервер)
| Крок | Хто | Дія |
|---|---|---|
| 1 | Клієнт | Перевіряє `canCreateTopic()`; оптимістично `spendBees(10)`; показує словник миттєво |
| 2 | Клієнт | Надсилає `create` / `applySync` на сервер |
| 3 | Сервер | Рахує наявні словники; якщо `count >= 2` → `spendBees(10)`; інакше 0; за `not_enough_bees` → 402 з полем `beeBalance` (`wallet.service.ts:44-52`) |
| 4 | Сервер | Повертає **підтверджений баланс** (у відповіді create/sync/`meta.beeBalance`) |
| 5 | Клієнт | Виставляє `SetBeeBalance(підтверджений)` — звірка, локальне оптимістичне значення затирається серверним |
| 6 | Клієнт | Якщо сервер відхилив (402 / обрізав понадлімітне) → відкат оптимістичної дії + банер NeedBees |

> **Дія для коду:** прибрати клієнтське списання з `createTopic` як «фінальне» (лишити оптимістичним зі звіркою), додати списання/валідацію квот у `applyClientSync`, додати поле підтвердженого балансу у відповіді create/sync. **Припущення (уточнити):** точна форма поля балансу у `TopicResponseDto` / `SyncResponseDto` — наразі його там немає.

## 5. [ЗАРАЗ] Вразливість `/wallet/rewarded-ad` → [НОВЕ] вимоги (D4)

### [ЗАРАЗ] Проблема
`POST /wallet/rewarded-ad` (`wallet.controller.ts:25-31`):
```
@Post('rewarded-ad')
async claimRewardedAd(@CurrentUser() user) {
  const row = await this.walletService.addBees(user.id, REWARDED_AD_BEE_AMOUNT); // +10
  return UserResponseDto.fromRow(row);
}
```
- **Немає верифікації**, що рекламу реально переглянуто.
- **Не ідемпотентний** — будь-який авторизований клієнт може слати «голий POST» у циклі і фармити +10 за виклик.

### [НОВЕ за D4] Вимоги
- **Server-Side Verification (SSV)** або **разовий nonce**: сервер видає nonce перед показом реклами, перевіряє підпис/коллбек рекламної мережі і нараховує **лише** за підтверджений перегляд.
- **Ідемпотентність:** повторний запит з тим самим nonce/transaction_id **не** нараховує вдруге.
- **Нарахування за рекламу заразом просуває лічильники ad-промо** (milestone «10 реклам», daily_streak «реклами/день», leaderboard «переглядів реклами/тиждень») — все це проходить через той самий верифікований ендпоінт, щоб промо-прогрес не можна було фармити. Деталі промо — файл **05-promo**.
- База **+10 лишається**; промо — бонус **зверху** (D4).

> **Припущення (уточнити):** яка рекламна мережа (AdMob SSV callback?) і чи зберігати спожиті nonce у БД vs кеш із TTL.

## 6. Критичний поріг ≤ 3 та стан «0 монеток»

| Стан | Умова | Реакція UI | Позначка |
|---|---|---|---|
| Нормальний | `beeBalance > 3` | звичайний badge балансу | [ЗАРАЗ] |
| Критичний | `beeBalance <= CriticalBeeThreshold` (≤ 3) | банер «майже закінчились монетки» на **не-домашніх** екранах | див. файл 05 |
| Порожній | `beeBalance == 0` | блокування платних дій; шторка NeedBees при спробі | [ЗАРАЗ] логіка гейтів |

- **`CriticalBeeThreshold = 3`** — `VocabeeStore.kt:29`. Поріг суто **клієнтський / UI-сигнал** (сервер про нього не знає).
- **CriticalBeeBanner** показується на **не-домашніх** екранах при `beeBalance <= 3`. Повний опис банерів (копірайт, де саме рендериться, пріоритети) — у файлі **05-promo**.
- **Стан «0 монеток»:** платні дії заблоковані гейтами — `canCreateTopic()` поверне `false` для 3-го словника (бо `beeBalance < 10`), `canSearchTranslation()` поверне `false` (бо `beeBalance < 1`). При спробі — шторка **NeedBees** (поповнити через рекламу/промо).
- Баланс не йде в мінус: `setBeeBalance`/`addBees`/`spendBees` застосовують `coerceAtLeast(0)` (`VocabeeStore.kt:357,366,373`), а на сервері `spendBees` має guard `gte(beeBalance, amount)` (`wallet.service.ts:39`) і кидає **402 `not_enough_bees`** (`wallet.service.ts:44-52`).

> **[НОВЕ]:** після переходу на серверний авторитет (D1) критичний/порожній стан клієнт малює за **підтвердженим** балансом (після `SetBeeBalance`), а не лише за оптимістичним.

## 7. Стани балансу і де вони показуються

| Стан | Де показується | Джерело даних | Позначка |
|---|---|---|---|
| Поточний баланс | badge монеток (іконка Sparkle/стільники) | `state.beeBalance` (`VocabeeStore.kt:41`) | [ЗАРАЗ] |
| Критичний (≤3) | CriticalBeeBanner на не-домашніх екранах + badge у банері | `beeBalance <= CriticalBeeThreshold` | банери — файл 05 |
| Недостатньо для дії | шторка **NeedBees** (поповнити рекламою/промо) | гейт повернув `false` / сервер 402 | [ЗАРАЗ] гейти; **[НОВЕ]** 402 від сервера |
| Порожній (0) | блокування платних дій, NeedBees при спробі | `beeBalance == 0` | [ЗАРАЗ] |

**Потік оновлення балансу в UI:**
- Локально-оптимістично: `AddBees` / `SpendBees` (`VocabeeStore.kt:158-159`) одразу змінюють `state.beeBalance` і пишуть у `preferencesManager.beeBalance`.
- Зі сервера-звіркою **[НОВЕ за D1]**: відповідь сервера (create / `/search` `meta.beeBalance` `RemoteLexiconSearchUseCase.kt:46`, sync, `applySync`) → `VocabeeEvent.SetBeeBalance` → `setBeeBalance()` (`VocabeeStore.kt:372-376`) виставляє **підтверджений** баланс.
- При вході в акаунт: `ApplyAuthenticatedAccount(... beeBalance ...)` (`VocabeeStore.kt:113-122`, обробник `:419-447`) тягне баланс із серверного профілю.

> **Примітка по `/search` [ЗАРАЗ]:** use-case `RemoteLexiconSearchUseCase` уже читає `response.meta.beeBalance` (`RemoteLexiconSearchUseCase.kt:46`) і повертає його в `Result.Ok.beeBalance`. Тобто канал для серверної звірки балансу **вже існує** на пошуку — лишається підключити його до `SetBeeBalance` і змусити сервер реально списувати −1 на `/search` (зараз списання на `/search` немає).
